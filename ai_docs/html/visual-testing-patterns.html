Plots, images, and tensors don't fit the assertEqual shape. You compare them in perceptual or numeric tolerance, not character-by-character. Here are the four pytest patterns worth knowing — with the code that makes them stick.

01

Matplotlib figures

pytest-mpl · image hashing · numeric data extraction

The mature answer is pytest-mpl: decorate the test, return a Figure, the plugin renders it and diffs against a stored baseline PNG. You generate the baseline once with --mpl-generate-path, check it in, and CI fails on visual drift.

tests/test_plots.py 
import matplotlib.pyplot as plt
import numpy as np
import pytest


@pytest.mark.mpl_image_compare(baseline_dir="baseline", tolerance=8)
def test_revenue_chart():
    fig, ax = plt.subplots(figsize=(6, 4))
    quarters = ["Q1", "Q2", "Q3", "Q4"]
    revenue  = [1.2, 1.8, 2.1, 2.6]
    ax.bar(quarters, revenue, color="#5cdbf0")
    ax.set_title("Revenue ($M)")
    return fig          # pytest-mpl picks this up


# Generate baseline once:
#   pytest --mpl-generate-path=tests/baseline
# Run normally:
#   pytest --mpl
Pitfall Baselines depend on the matplotlib version, the backend, freetype, and the OS font stack. Pin them. The community pattern is a fixed Docker image (or uv lock) plus matplotlib.use("Agg") in conftest.py. Don't generate baselines on macOS and run CI on Linux.

When you don't want pixel comparison at all — sometimes you just want to assert the right numbers got plotted — interrogate the Axes directly. This is faster, more stable, and tells the next reader what the test actually cares about.

tests/test_plot_data.py 
def test_bar_heights_match_input():
    fig, ax = plt.subplots()
    ax.bar(["a", "b", "c"], [1, 2, 3])

    heights = [bar.get_height() for bar in ax.patches]
    assert heights == [1.0, 2.0, 3.0]
    assert ax.get_xticklabels()[0].get_text() == "a"


def test_line_data_matches():
    fig, ax = plt.subplots()
    ax.plot([0, 1, 2], [0, 1, 4], label="x²")

    line = ax.get_lines()[0]
    np.testing.assert_array_equal(line.get_xdata(), [0, 1, 2])
    np.testing.assert_array_equal(line.get_ydata(), [0, 1, 4])
    assert line.get_label() == "x²"
Heuristic Pixel tests for "does this look right". Data extraction for "is the math right". Most plotting bugs are math bugs disguised as rendering bugs — go data-first.
02

OpenCV & Pillow image diffing

absdiff · SSIM · perceptual hash · golden masters

Three tools cover almost everything. cv2.absdiff gives you a raw pixel-difference mask, scikit-image.metrics.structural_similarity (SSIM) gives a perceptual score that tolerates compression noise, and imagehash gives a hash you can compare with a Hamming distance threshold — useful when image dimensions or formats drift.

pixel diff → mask → similarity score → assertion

Approach A — OpenCV absdiff. Loud, exact, intolerant. Use for rendered output where every pixel should match (icons, generated thumbnails on the same pipeline).

tests/test_images_cv.py 
import cv2
import numpy as np
import pytest
from pathlib import Path


@pytest.fixture
def expected():
    return cv2.imread("tests/fixtures/expected.png")


def test_thumbnail_matches(expected, tmp_path: Path):
    actual = my_pipeline.render_thumbnail()         # your code under test

    assert actual.shape == expected.shape, "size mismatch"
    diff = cv2.absdiff(actual, expected)
    if diff.any():
        # On failure, drop the diff so you can eyeball it in the artifact
        cv2.imwrite(str(tmp_path / "diff.png"), diff * 10)
        cv2.imwrite(str(tmp_path / "actual.png"), actual)
        print(f"diff written to {tmp_path}")

    # Tolerance: average per-channel delta < 1 (out of 255)
    assert diff.mean() < 1.0

Approach B — SSIM via scikit-image. The right call when the image went through any lossy step (JPEG, resize, anti-aliasing differences). Returns a score in [-1, 1]; ≥0.95 is usually "looks the same".

tests/test_images_ssim.py 
from skimage.metrics import structural_similarity as ssim
from PIL import Image
import numpy as np


def _gray(path):
    return np.asarray(Image.open(path).convert("L"))


@pytest.mark.parametrize("name,threshold", [
    ("logo.png",      0.98),   # crisp asset
    ("hero_jpg.jpg",  0.92),   # jpeg, lossier
])
def test_processed_image_perceptually_matches(name, threshold):
    expected = _gray(f"tests/fixtures/{name}")
    actual   = _gray(f"build/out/{name}")

    score, diff = ssim(expected, actual, full=True)
    assert score >= threshold, f"SSIM {score:.3f} < {threshold}"

Approach C — Perceptual hashing. Cheap, robust to resize / re-encode / format change. The right call when you just want to say "is this still essentially the same image" across formats.

tests/test_images_phash.py 
import imagehash
from PIL import Image


def test_thumbnail_essentially_unchanged():
    a = imagehash.phash(Image.open("tests/fixtures/cover.png"))
    b = imagehash.phash(Image.open("build/cover.webp"))

    # Hamming distance: 0 = identical, <=5 = visually identical for phash
    assert (a - b) <= 5

One pattern worth pinching: the golden-master with auto-update. Save the actual output next to the baseline on failure, and add a --update-baselines flag your team uses deliberately. That makes reviews "is the new image correct?" instead of "why did this test fail again?".

tests/conftest.py 
def pytest_addoption(parser):
    parser.addoption(
        "--update-baselines",
        action="store_true",
        help="Overwrite golden images with current output.",
    )

@pytest.fixture
def update_baselines(request):
    return request.config.getoption("--update-baselines")


# in your test:
def test_render(update_baselines, tmp_path):
    actual = my_render()
    baseline = Path("tests/baselines/render.png")
    if update_baselines:
        cv2.imwrite(str(baseline), actual)
        pytest.skip("baseline updated")
    expected = cv2.imread(str(baseline))
    assert cv2.absdiff(actual, expected).mean() < 1.0
03

pytest-mock — the mocker fixture

patching · spying · auto-cleanup · I/O isolation

Yes, pytest-mock is almost always involved in visual/image tests — because the subject under test usually reads from disk, calls an API, or hits a model server, and you don't want any of that running in unit tests. It's a thin pytest wrapper over unittest.mock that gives you a mocker fixture with automatic teardown.

without pytest-mock

from unittest import mock

def test_x():
    with mock.patch(
        "app.io.load_image",
        return_value=fake_img,
    ):
        with mock.patch("app.io.save"):
            run()

with pytest-mock

def test_x(mocker):
    mocker.patch(
        "app.io.load_image",
        return_value=fake_img,
    )
    mocker.patch("app.io.save")
    run()
tests/test_pipeline.py 
import numpy as np


def test_pipeline_calls_save_with_processed_array(mocker):
    # 1. Stub the loader so no disk access
    fake = np.zeros((64, 64, 3), dtype=np.uint8)
    mocker.patch("app.io.load_image", return_value=fake)

    # 2. Spy on save: real impl can run, but we get a call record
    save_spy = mocker.spy(app.io, "save")

    # 3. Patch a class method on an instance the code constructs internally
    mocker.patch.object(
        app.models.Detector,
        "infer",
        return_value={"boxes": [], "scores": []},
    )

    app.pipeline.run("in.png", "out.png")

    assert save_spy.call_count == 1
    args, kwargs = save_spy.call_args
    assert args[0] == "out.png"
    assert args[1].shape == (64, 64, 3)


def test_http_failure_is_handled(mocker):
    # Side effect = raise; useful for testing error paths
    mocker.patch(
        "app.client.requests.get",
        side_effect=ConnectionError("boom"),
    )
    result = app.client.fetch_image("http://x")
    assert result is None
Rule of thumb mocker.patch replaces, mocker.spy wraps. Use spy when you want the real thing to run AND want to assert on how it was called. Use patch when calling the real thing is slow, networked, or unsafe.

One subtlety that bites people: patch where the name is looked up, not where it's defined. If app/pipeline.py does from app.io import save, you patch app.pipeline.save, not app.io.save. This single confusion accounts for most "the patch isn't working" tickets.

04

pytest-pytorch

selecting tests by name in PyTorch's parametrized suite

This one's narrower than the others. PyTorch's test suite uses its own runtime parametrization (templates instantiated per device / dtype), and the generated names don't match the names you see in source. So when you try to run a single test by name — especially from the green-arrow next to a test in VSCode or PyCharm — it doesn't resolve.

What it actually does The plugin hooks pytest's collection phase and maps the names-as-written back to PyTorch's generated names, so the standard pytest selection syntax just works. No code changes in your tests — install it and the green arrow starts behaving.
shell 
# Install
pip install pytest-pytorch
# or: conda install -c conda-forge pytest-pytorch

# Before — fails or runs nothing:
pytest test_foo.py::TestBar::test_baz

# After — runs test_baz against every instantiated device:
pytest test_foo.py::TestBar::test_baz

# Narrow to one device with -k (pytest-mark-parametrize style):
pytest test_foo.py::TestBar::test_baz -k cpu
pytest test_foo.py::TestBar::test_baz -k cuda

If you're not actively contributing to PyTorch itself, you probably don't need it. The place it shines is when you've cloned pytorch/pytorch, opened test/test_torch.py, and want to run one method via your IDE without memorising the auto-generated suffix scheme.

For testing your own code that uses PyTorch, lean on stock pytest plus torch.testing.assert_close for tensor comparison — it handles dtype, device, and tolerance properly so you don't have to remember rtol/atol defaults per use case.

tests/test_my_model.py 
import torch
import pytest


@pytest.mark.parametrize("device", ["cpu", pytest.param("cuda",
    marks=pytest.mark.skipif(not torch.cuda.is_available(), reason="no gpu"))])
def test_my_layer_output_shape(device):
    layer = MyLayer(in_dim=16, out_dim=32).to(device)
    x = torch.randn(4, 16, device=device)
    y = layer(x)
    assert y.shape == (4, 32)


def test_my_layer_matches_reference():
    torch.manual_seed(0)
    out      = MyLayer(8, 8)(torch.ones(2, 8))
    expected = torch.tensor([[0.31] * 8, [0.31] * 8])
    torch.testing.assert_close(out, expected, rtol=1e-2, atol=1e-2)
05

When to reach for what

decision table
You're testingReach forWhy
A matplotlib chart looks correct pytest-mpl Baseline-PNG comparison with tolerance; first-class plugin.
The right data got plotted plain pytest + Axes inspection Faster, more stable, more diagnostic than pixel diff.
Exact pixel match (icons, rendered HTML, generated thumbnails) cv2.absdiff Loud and exact; great for deterministic pipelines.
Perceptual match (JPEG, ML output, anti-aliased) skimage.metrics.ssim Tolerates compression and minor numerical drift.
"Roughly the same image" across formats / sizes imagehash.phash Cheap; format- and size-agnostic; integer threshold.
Code that touches disk, network, or a model pytest-mock mocker.patch, mocker.spy, auto-cleanup.
Tensors, your code torch.testing.assert_close Handles device, dtype, rtol/atol cleanly.
PyTorch's own test suite (contributing upstream) pytest-pytorch Makes IDE green-arrow selection work with PyTorch's parametrization.
